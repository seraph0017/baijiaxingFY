# 百家姓溯源录上线 Review 报告

## 当前结论

当前版本已经从静态页面补成 Node.js 网站 MVP，具备可部署的服务入口、前端产品界面、前台/运营台分区、AI Harness、资料沉淀、审核队列、纠错反馈落库、运行数据持久化、审计日志、公开站上线资产、Docker 部署文件和 release 级验收脚本。

## 已完成能力

- 网站前台：姓氏查询、姓氏档案、源流分支、迁徙路线、望族分支、图腾字形、典籍来源、历史名人与家风文化。
- 公开查询 API：`GET /api/surnames`、`GET /api/surnames?q=chen&limit=20`、`GET /api/surname?name=陈`，便于后续小程序或独立前端复用。
- 产品分区：普通用户前台与运营工作台分离。
- AI Harness：Markdown 上下文检索、OpenAI-compatible 接口代理、无 Key 离线初稿、审核队列沉淀。
- 资料沉淀：新增 Markdown 资料、导入导出 JSON、服务端工作区保存、历史备份。
- 档案编辑：运营台可人工校订当前姓氏基础字段和摘要，保存后同步前台展示并进入审核队列。
- 反馈管理：纠错反馈写入 `feedback.jsonl`，运营台可查看并标记处理中/已处理/已关闭，所有处理动作进入审计日志。
- 审计事件：运营台可查看服务端审计事件，追踪资料保存、反馈处理和 AI 初稿请求。
- 管理令牌验证：运营台可即时验证管理令牌，并自动刷新反馈工单与审计事件。
- 数据边界：公开前台通过公开姓氏 API 初始化，完整工作区读取需要管理令牌。
- 服务端：`/api/health`、`/api/bootstrap`、`/api/workspace`、`GET /api/audit`、`GET/POST/PATCH /api/feedback`、`/api/ai-draft`。
- 上线基础：管理令牌、请求限流、AI 超时、HTTP 请求超时、安全响应头、审计日志、运行数据目录可配置。
- 公开站上线资产：SEO 元信息、分享卡片元信息、`robots.txt`、`sitemap.xml`、`manifest.webmanifest`，并支持 `SITE_ORIGIN` 生成正式域名地址。
- 部署：`Dockerfile`、`.dockerignore`、`.env.example`、生产部署说明、容器健康检查。
- 验收：静态验收、UI 结构验收、无端口服务逻辑验收、release 一键验收。
- 鉴权验收：无端口验证管理令牌启用后，后台读写接口拒绝无令牌请求并接受正确令牌请求。

## 本轮 Review 发现并修复的问题

1. 原实现依赖真实端口验收，当前受限环境无法证明服务逻辑。
   - 修复：新增 `verify-server-logic.mjs`，用 mock request/response 直接测试 `handleRequest`。

2. `server.js` 导入即监听端口，不利于测试和复用。
   - 修复：导出 `handleRequest` 和 `createAppServer`，仅直接运行 `node server.js` 时监听端口。

3. Docker 挂载 `/app/data` 会覆盖内置种子资料。
   - 修复：新增 `DATA_DIR`，运行数据默认可配置；镜像内置 `data/seed-workspace.json` 不被运行卷覆盖。

4. 上线缺少超时控制和存储可写性检查。
   - 修复：新增 `AI_TIMEOUT_MS`、`REQUEST_TIMEOUT_MS`、`storageWritable`、`runtimeDirConfigured`。

5. 缺少提交前统一验收命令。
   - 修复：新增 `npm run verify:release`，串行跑语法、静态、UI、服务逻辑验收。

6. 后台能力混在用户浏览路径里，产品边界不清晰。
   - 修复：新增前台/运营台分区切换，把 AI Harness、审核队列、资料导入导出放入运营工作台。

7. 纠错反馈只做前端模拟，无法形成运营闭环。
   - 修复：新增 `POST /api/feedback`，服务端写入 `feedback.jsonl` 并记录 `feedback.create` 审计事件。

8. 反馈落库后运营台不可见，仍缺处理闭环。
   - 修复：新增 `GET/PATCH /api/feedback` 和运营台反馈工单区，支持读取、处理中、已处理、关闭。

9. 姓氏档案只能由 AI 或 JSON 导入更新，缺人工校订入口。
   - 修复：新增运营台姓氏档案编辑器，可校订繁体、拼音、起源朝代、始祖、郡望、堂号和源流摘要。

10. 审计日志只能落文件，运营台不可见。
   - 修复：新增 `GET /api/audit` 和运营台审计事件区，支持查看最近操作事件。

11. 公开网站缺少正式上线常用资产，搜索引擎、分享卡片和移动端保存入口不完整。
   - 修复：新增 SEO / Open Graph / Twitter Card 元信息，补齐 `robots.txt`、`sitemap.xml`、`manifest.webmanifest`，并由 `SITE_ORIGIN` 控制公开地址。

12. 运营人员无法提前判断管理令牌是否正确，只能在刷新反馈或审计失败后倒推问题。
   - 修复：新增运营台“验证管理令牌”动作，验证通过后自动刷新反馈工单与审计事件。

13. 管理令牌鉴权只有代码实现和正则检查，缺少 release 级行为证据。
   - 修复：新增 `verify-server-auth.mjs`，无端口验证后台读写接口 401/200 路径，并纳入 `npm run verify:release`。

14. 姓氏数据只通过 bootstrap 和前端内部状态使用，后续小程序缺少稳定公开查询接口。
   - 修复：新增 `GET /api/surnames` 和 `GET /api/surname?name=陈`，统一读取运行工作区或种子资料库。

15. 公开姓氏列表缺少搜索和数量限制，姓氏库扩展后列表页会一次拉取过多数据。
   - 修复：`GET /api/surnames` 新增 `q` 与 `limit` 参数，返回 `query`、`limit`、`total` 和截断后的 `surnames`。

16. 完整工作区读取未做运营鉴权，可能把 Markdown 语料和审核队列作为公开接口暴露。
   - 修复：`GET /api/workspace` 在设置 `ADMIN_TOKEN` 后必须携带管理令牌；前端初始化改为公开姓氏 API 优先。

17. Docker 镜像缺少容器级健康检查，容器平台无法自动感知实例异常。
   - 修复：Dockerfile 新增 `HEALTHCHECK`，定期请求 `/api/health`。

18. Docker 镜像未复制 `robots.txt`、`sitemap.xml`、`manifest.webmanifest`，容器部署后公开站资产会返回 404。
   - 修复：Dockerfile 显式复制三个公开站根文件。

19. 运营台“清空本地数据”会静默吞掉后端清空失败，容易让运营误以为服务端工作区已清。
   - 修复：清空动作改为异步检查后端响应；失败时明确提示“本地已清、后端未清”，并提示检查服务状态或管理令牌。

20. 工作区保存只校验 `surnames` 是对象，未校验每个姓氏档案结构，坏数据可能进入公开查询 API。
   - 修复：服务端校验每个姓氏条目必须是对象，且 `char` 必须与键名一致。

21. 运行工作区文件损坏时，公开姓氏查询 API 会直接失败，前台页面可能被运营数据文件拖垮。
   - 修复：公开查询路径读取运行工作区失败时自动回退到种子库；完整工作区运营读取仍暴露错误，便于排查。

22. `feedback.jsonl` 或 `audit.log` 出现单行坏 JSON 时，运营台读取会整体失败。
   - 修复：新增 JSONL 安全解析，坏行跳过，正常反馈和审计记录继续可读。

23. 工作区保存直接覆盖 `workspace.json`，进程中断时可能留下半截 JSON。
   - 修复：保存改为同目录临时文件写入后 `rename` 原子替换，降低运行文件损坏概率。

24. 反馈工单状态更新会重写 `feedback.jsonl`，中断时也可能留下半截队列文件。
   - 修复：反馈队列写回同样改为临时文件写入后 `rename` 原子替换。

25. 前台和详情 API 会把输入姓氏截成 1 个字，无法正确支持“欧阳”等复姓。
   - 修复：前端统一 `normalizeSurnameInput`，服务端详情查询支持最多 4 字姓氏。

26. 纠错反馈前台没有联系方式入口，运营处理工单时无法追问线索。
   - 修复：前台新增选填联系方式字段，提交反馈时同步写入服务端反馈记录。

27. 反馈联系方式虽已提交到服务端，但运营反馈工单台没有展示，处理人员仍然看不到追问入口。
   - 修复：运营反馈列表增加“联系方式”展示；未填写时自动隐藏，已填写时使用转义渲染。

28. 静态资源路径遇到错误 URL 编码时可能表现为服务端异常，不利于上线后排查。
   - 修复：静态资源解码失败时明确返回 400，避免把异常请求误判为服务故障。

29. AI Harness 只有浏览器里填写 API Key 才会调用 AI，和“生产走服务端环境变量”的技术口径不一致。
   - 修复：Harness 改为优先调用服务端 AI 代理；服务端已配置环境变量时无需在前端暴露 Key，失败时再回退离线初稿。

30. README 的 AI Harness 说明仍停留在“前端填写 Key 才调用”的旧口径，和最新实现不一致。
   - 修复：README 同步为“优先服务端 AI 代理、前端 Key 仅本地调试、失败回退离线初稿”，并加入静态验收。

31. 服务端 AI 代理会优先使用前端传入的 endpoint/model，可能覆盖生产环境配置的企业代理地址或指定模型。
   - 修复：`AI_ENDPOINT`、`AI_API_KEY`、显式 `AI_MODEL` 优先于前端 payload；前端字段只作为本地临时调试 fallback。

32. 工作区导入使用合并语义，导入新资料库时旧姓氏不会被移除，容易让运营误以为已切换到干净数据集。
   - 修复：快照中包含 `surnames` 时改为覆盖当前姓氏库，避免历史脏数据残留。

33. 未入库姓氏只能靠用户逐个搜索触发待收录，不利于运营按高频姓氏名单批量沉淀样板库。
   - 修复：运营台新增“批量补充姓氏”，支持空格、逗号、顿号或换行分隔，一次性创建待收录档案并加入审核队列。

34. 页面 Harness 说明仍写“未填写 Key 时走离线”，和最新“优先服务端 AI 代理”实现不一致，演示时容易误导业务方。
   - 修复：页面文案同步为服务端 AI 代理优先，失败时回退离线样例。

35. 管理令牌会保存在当前浏览器会话中，但运营台没有清除入口，演示或共享设备上容易留下权限痕迹。
   - 修复：运营台新增“清除令牌”，同步清空输入框、会话令牌、反馈队列和审计队列显示。

36. 用户输入“陈姓”“欧阳姓”时，前台会把它当作新姓氏创建待收录档案，服务端详情 API 也会查不到正式档案。
   - 修复：前端和服务端统一去除末尾“姓”后再查询，兼容单姓和复姓常见输入习惯。

37. 静态资源白名单先按 `/assets/` 判断，再解码路径；编码斜杠可能让请求跳出 `assets` 目录读取根目录文件。
   - 修复：新增真实 `assets` 目录边界校验，解码后的资源路径必须仍位于 `assets` 目录内。

38. 详情 API 已兼容“陈姓/欧阳姓”，但公开列表搜索 `q=欧阳姓` 仍搜不到“欧阳”，会影响后续小程序搜索复用。
   - 修复：`GET /api/surnames` 搜索词同步生成去“姓”后缀的匹配词，单姓和复姓列表搜索体验一致。

39. 公开站没有 favicon 和 Manifest 图标，上线后浏览器标签页、收藏入口和安装入口会显得像默认 demo。
   - 修复：新增红金印章风 `assets/icon.svg`，接入 HTML favicon 与 Web App Manifest icons。

40. 反馈工单 ID 只使用 `Date.now()`，同毫秒提交多条反馈时可能重复，后续状态更新会命中错误工单。
   - 修复：反馈 ID 增加 `randomUUID` 短随机后缀，并补充同毫秒提交不重复验收。

41. 已兼容“陈姓/欧阳姓”，但姓氏文化语境下用户也常输入“陈氏/欧阳氏”，前台和公开 API 仍会查不到或创建错误待收录。
   - 修复：前端、详情 API、列表搜索统一去除末尾“姓/氏”后再处理。

42. 公开反馈接口没有复用姓氏归一化，外部前端提交“欧阳氏”会在运营台显示为“欧阳氏姓反馈”，影响工单聚合。
   - 修复：`POST /api/feedback` 的 `surname` 同步去除末尾“姓/氏”后保存。

43. 生产环境如果忘记设置 `ADMIN_TOKEN`，后台读写和 AI 代理接口会按本地演示模式开放，存在严重误配置风险。
   - 修复：`NODE_ENV=production` 且缺少 `ADMIN_TOKEN` 时，后台读写和 AI 代理接口返回 503；新增 `npm run verify:production` 并纳入 release 验收。

44. 虽然后台接口已拒绝生产缺令牌，但 `/api/health` 仍返回 200，Docker 健康检查会把误配置实例标记为 healthy。
   - 修复：生产缺少 `ADMIN_TOKEN` 时 `/api/health` 返回 503，并带 `configError`，容器健康检查可正确失败。

45. 生产环境如果设置了 `ADMIN_TOKEN` 但漏配 `AI_ENDPOINT` / `AI_API_KEY`，AI 代理仍可能使用前端传入 Key/Endpoint 兜底，和生产配置口径不一致。
   - 修复：生产环境强制配置 `AI_ENDPOINT` 和 `AI_API_KEY`；缺失时 health 与 AI 代理均返回 503。

46. 生产环境使用服务端 `AI_MODEL` 覆盖前端模型时，AI 审计日志仍记录前端 payload 的 `model`，导致台账和真实调用不一致。
   - 修复：抽出 `resolveAiConfig`，AI 调用和审计日志共用同一份实际配置。

47. README 生产建议仍写“缺少 ADMIN_TOKEN 时写接口保持本地演示模式”，和最新生产强制令牌策略冲突。
   - 修复：README 改为生产缺令牌时后台接口拒绝服务、健康检查返回 503，并加入静态验收防回退。

48. README 的直接 Node 部署命令未设置 `NODE_ENV=production`，会绕过生产配置检查，只 Docker 路径具备生产安全策略。
   - 修复：直接部署命令补齐 `NODE_ENV=production`、AI 服务端环境变量，`.env.example` 同步增加 `NODE_ENV=production`。

49. AI 代理接口收到 `null` 或数组这类非对象 JSON 时，会进入配置解析并触发服务端异常，不利于外部调用方定位请求格式问题。
   - 修复：`resolveAiConfig` 增加请求体对象校验，非对象请求明确返回 400，并纳入无端口逻辑验收。

50. 前台初始化只读取 `/api/surnames?limit=100`，当资料库沉淀超过 100 个姓氏后，首页搜索和小程序复用列表会漏掉后续档案。
   - 修复：公开列表单次上限提升到 500，前端初始化改为读取 500 条，并补充 UI 与服务端逻辑验收。

51. AI 代理只判断 `messages` 是数组，不判断消息条目结构；外部客户端传入字符串数组时会打到模型接口，调用方只能看到上游失败。
   - 修复：新增 `validateAiMessages`，要求 `messages` 非空且每条包含合法 `role` 与非空 `content`，错误请求直接返回 400。

52. 公开详情接口会原样返回运营导入的半成品姓氏档案；如果缺少 `tags/origins/migrations/route/visuals/figures/sources` 等字段，前端渲染可能白屏。
   - 修复：新增 `normalizePublicSurnameProfile`，公开列表和详情统一补齐可渲染默认结构，半成品资料仍可安全展示为“待补来源”状态。

53. `/api/health` 虽然会返回 `storageWritable`，但状态码只看生产配置；如果运行目录不可写，容器健康检查仍可能返回 200。
   - 修复：健康状态改为同时依赖生产配置和运行目录可写性；`DATA_DIR` 不可写时返回 503，并纳入生产安全验收。

54. 外部客户端可以提交没有姓氏的纠错反馈，运营台会出现“未指定姓反馈”，无法聚合到具体姓氏资料沉淀链路。
   - 修复：`POST /api/feedback` 要求归一化后的 `surname` 非空，空姓氏反馈直接返回 400。

55. 服务端公开详情已能补齐半成品档案，但运营台 JSON 导入和 localStorage 恢复仍直接使用原始姓氏对象，半成品数据可能导致前端渲染白屏。
   - 修复：前端新增 `normalizeClientSurnameProfile`，工作区导入、公开 API 初始化和本地恢复统一补齐可渲染默认结构。

56. 工作区 schema 只校验 Markdown 语料和审核队列字段存在，不校验字段类型；对象或数字可能进入长期资料库，影响检索、展示和后续小程序复用。
   - 修复：`markdownCorpus.surname/title/content` 与 `reviewState.surname/title/status` 必须是非空字符串，类型错误返回 400。

57. `feedback.jsonl` 中如果存在历史脏状态，运营台会原样展示不可控状态标签，影响工单流转和统计。
   - 修复：新增 `normalizeFeedbackItem`，反馈列表读取时统一归一化状态；非法状态回退为“待处理”。

58. 历史反馈如果缺少 `id`，读取时会每次生成随机导入 ID，运营台刷新后同一工单的定位会变化，后续状态更新难以追踪。
   - 修复：新增 `stableFeedbackId`，按姓氏、内容和创建时间生成稳定哈希导入 ID。

59. 审核队列批准/驳回使用 `title` 定位条目，多个同名 AI 初稿或同名资料时可能更新错审核项。
   - 修复：新增 `createReviewItem`、`normalizeReviewItem` 和 `stableReviewId`，审核操作改为按稳定 `id` 更新。

113. 405 响应缺少 `Allow` 头，客户端和代理无法直接知道该路径支持哪些 HTTP 方法。
   - 修复：API 路由改为维护路径到允许方法的映射；已知 API 方法不匹配返回对应 `Allow`，静态资源 405 返回 `Allow: GET, HEAD`，逻辑验收覆盖 API 和静态资源两类 405。

114. API 兜底的 `HEAD` 请求虽然能返回正确状态码和 `Allow`，但仍返回 JSON 正文，不符合 HEAD 响应不应包含 body 的语义。
   - 修复：API 前缀兜底在 `HEAD` 请求下只返回响应头，不返回正文；原 405/404 状态和 `Allow` 头保持不变，逻辑验收覆盖 API HEAD 兜底空正文。

115. 请求入口的 `new URL(req.url, host)` 在统一 `try/catch` 外执行，非法 `Host` 头会让 `handleRequest` 直接 reject，无法返回结构化错误响应。
   - 修复：新增 `parseRequestUrl`，把 URL 解析纳入统一错误处理；非法 Host 或请求 URL 返回 400「请求 URL 不正确」，无端口逻辑验收覆盖非法 Host 不再导致请求处理直接抛出。

116. README 说明镜像内可执行 `npm run verify:release`，但 Dockerfile 默认 `NODE_ENV=production`，如果没有显式传入生产必需环境变量，容器内 release 自检会因为健康检查配置不完整而失败。
   - 修复：README 增加可直接复制的 `docker run --rm ... npm run verify:release` 命令，带齐 `ADMIN_TOKEN`、`AI_ENDPOINT`、`AI_API_KEY` 和运行数据卷；静态上线验收覆盖该文档示例。

117. 错误响应缺少请求 ID，线上排查时用户截图、代理日志和服务端日志难以关联到同一次请求。
   - 修复：请求进入时生成或复用合法 `x-request-id`；所有响应带 `x-request-id` 响应头，JSON 错误体同步返回 `requestId`，无端口逻辑验收覆盖非法 Host 错误响应的请求 ID。

118. 审计日志没有记录请求 ID，即使错误响应已经带 `requestId`，运营台事件仍难以和用户请求、代理日志关联。
   - 修复：`assignRequestId` 同步写入 `req.requestId`，`appendAudit` 将 `requestId` 落到每条审计记录；无端口逻辑验收覆盖 `workspace.save` 审计事件包含请求 ID。

119. 审计事件虽然已经落入请求 ID，但运营台仍把它藏在详情 JSON 里，业务同学排查反馈单时不方便复制给技术。
   - 修复：运营审计列表独立展示“请求 ID”，老事件缺失时自动隐藏；样式允许长 ID 断行，避免撑破审计卡片；UI 验收覆盖该字段可见性。

120. 静态资源响应缺少 `Content-Length`，`HEAD` 请求也无法返回与 `GET` 一致的内容长度，代理缓存和健康探测不够标准。
   - 修复：`send()` 统一计算响应体长度；静态资源在 `GET/HEAD` 中复用真实文件内容长度；无端口逻辑验收覆盖静态 `GET` 与 `HEAD` 内容长度一致。

121. API JSON 响应虽然已经由统一 `send()` 写入 `Content-Length`，但缺少直接验收覆盖，后续修改响应函数时可能只保护静态资源、不保护 API。
   - 修复：无端口逻辑验收新增 `/api/health` JSON 响应内容长度校验，确保 API 响应头与实际 UTF-8 字节数一致。

122. 公开 API 支持 `q=chen` 这类拼音检索，但前台主搜索框只按字面姓氏处理；用户输入 `chen` 会被创建成新的待收录档案，演示和真实使用都容易误导。
   - 修复：新增 `resolveSurnameQuery`，前台搜索优先按简体、繁体、`pinyin` 和 `info["拼音"]` 精确匹配已沉淀姓氏；匹配不到时再按原逻辑创建待收录档案；UI 验收覆盖拼音查询入口。

123. 拼音查询只接入了搜索按钮，新增 Markdown 资料、AI Harness、纠错反馈和档案编辑仍通过旧的 `getCurrentSurname()` 读取输入框；如果用户输入 `chen` 后直接操作，后台仍会沉淀到错误的“chen姓”。
   - 修复：`getCurrentSurname()` 统一复用 `resolveSurnameQuery`，所有依赖当前姓氏的前台和运营动作都按简体、繁体、拼音解析到已沉淀档案；UI 验收覆盖当前姓氏解析一致性。

124. 已收录拼音能命中档案，但未收录纯英文输入如 `zhang` 会被截断成 `zhan` 并创建待收录档案，污染姓氏沉淀库。
   - 修复：新增 `isLatinQuery`，纯英文输入只有匹配到已收录拼音时才命中档案；未命中时回退默认样板，不创建拉丁字母姓氏档案；UI 验收覆盖该防污染规则。

125. 部署探针或网关如果使用 `HEAD /api/health` 做轻量健康检查，旧实现会返回 405；虽然 Docker 自带健康检查用 GET，但部分平台仍会把 HEAD 405 当作实例不可用。
   - 修复：`/api/health` 支持 `GET/HEAD`，HEAD 只返回状态和响应头不返回正文；`POST /api/health` 仍返回 405 且 `Allow: GET, HEAD`；无端口逻辑验收覆盖部署探针兼容路径。

126. 已防住纯英文未命中拼音，但 `zhang1`、`chen.` 这类带数字或标点的拉丁输入仍会绕过，可能被截断后写入姓氏库。
   - 修复：将前端污染判断扩展为 `isLatinLikeQuery`，覆盖拉丁字母、数字、空格、点、下划线和横线；这类输入只有命中已收录拼音才使用，否则回退默认样板；UI 验收覆盖混合输入防污染规则。

127. 批量补充姓氏入口仍直接 `normalizeSurnameInput` 每个 token，运营粘贴名单里混入 `zhang1`、`wang_test` 时会创建被截断的错误待收录档案。
   - 修复：`parseBatchSurnames` 在归一化前过滤 `isLatinLikeQuery` 命中的拉丁/数字/标点混合 token，只允许汉字姓氏进入批量待收录流程；UI 和静态验收覆盖批量入口防污染。

128. 前端已过滤拉丁混合姓氏，但公开 `POST /api/feedback` 可被直接调用，`surname=zhang1` 仍会归一化成 `zhan` 写入反馈队列，污染运营聚合。
   - 修复：服务端新增 `isLatinLikeName`，新反馈提交遇到拉丁/数字/标点混合姓氏直接返回 400「反馈姓氏格式不正确」；历史反馈读取仍保留容错归一化；无端口逻辑验收覆盖公开写接口。

129. `POST /api/workspace` 只校验 `char === key`，直接调用接口或导入 JSON 时仍可保存 `zhan`、`zhang1`、`wang_test` 这类拉丁混合姓氏，污染长期资料库主表、Markdown 语料和审核队列。
   - 修复：服务端工作区保存统一拒绝拉丁/数字/标点混合的 `surnames` 键名、`char`、`markdownCorpus[].surname` 和 `reviewState[].surname`；前端导入 JSON 入口同步拦截同类脏数据；无端口逻辑验收和 UI 验收覆盖主表、资料、审核三条路径。

130. 写接口触发 429 限流时只返回错误文案，没有 `Retry-After`，前端、脚本或后续小程序无法按服务端窗口做退避重试。
   - 修复：限流错误按当前桶剩余窗口秒数返回 `Retry-After` 响应头；错误响应透传安全白名单式附加头；鉴权验收覆盖 429 响应头。

131. 运营台导入 JSON 只严格校验 `surnames`，但 `markdownCorpus` 或 `reviewState` 若是对象、空字段或缺字段，会先进入浏览器本地工作区，再由服务端保存失败，造成前端展示与后端状态不一致。
   - 修复：前端导入入口同步校验 `markdownCorpus` 和 `reviewState` 类型；资料条目必须包含非空字符串 `surname/title/content`，审核条目必须包含非空字符串 `surname/title/status`；坏快照在本地入口直接失败，UI 验收覆盖。

132. `/assets/` 目录路径本身命中静态资源白名单后会尝试读取目录，底层错误可能把服务器文件路径带回响应。
   - 修复：静态资源存在后必须通过 `statSync(filePath).isFile()`，目录或其他非文件路径直接返回 404；无端口逻辑验收覆盖 `/assets/` 不泄漏项目路径。

133. 当前页面仍偏米白运营工作台，与业务方给的深色科技国风参考图不一致，首页缺少强搜索入口和姓氏宫格，详情页缺少大号姓氏身份卡。
   - 修复：前台视觉重构为深色资料库风格：首页保留产品名并加入“AI 可信资料网络”主视觉、宽搜索框、姓氏宫格；详情页加入返回首页、大号姓氏身份卡、动态姓氏水印、深色 Tab 与档案卡片；运营台功能保留并统一暗色面板。

134. 迁徙地图节点依赖 `style` 内联定位，导致 CSP 必须保留 `style-src 'unsafe-inline'`，上线安全头不够收紧。
   - 修复：迁徙节点改为 `data-route-x/y` + 量化 class 定位，渲染后由 `positionRouteNodes()` 添加 class；服务端 CSP 收紧为 `style-src 'self'`，静态/UI 验收覆盖无内联样式依赖。

135. 参考图首页姓氏宫格展示单字姓氏，当前按钮仍显示“陈姓”；详情页大字水印 CSS 读取 `.profile-head[data-surname]`，但 JS 写到了外层 `#profile`，导致水印不稳定。
   - 修复：热门姓氏宫格改为只展示姓氏本字；`renderSurname` 将 `data-surname` 写入 `.profile-head`，确保详情身份卡动态水印生效；UI 验收覆盖单字宫格和水印绑定。

136. 详情身份卡虽然有大号姓氏与水印，但缺少参考图里“拼音/关键线索/来源数/审核状态”的一眼可信信息层，用户进入详情后仍要下翻才能判断资料完整度。
   - 修复：新增 `profileMeta` 可信信息胶囊，展示拼音、始祖线索、来源条数和审核状态；来源数从当前档案 `sources` 计算，审核状态从审核队列判断；UI 和静态验收覆盖。

137. 胶囊状态只看审核队列是否有“已发布”，种子样板有来源但未发布时统一显示“待审核校订”，容易让前台用户误以为样板内容完全未沉淀。
   - 修复：新增 `resolveProfileReviewStatus`，状态分为“已审核发布 / 来源待核 / 待收录 / 待补来源”；有来源但未发布的样板显示“来源待核”，待收录档案明确显示“待收录”；UI 与静态验收覆盖。

138. 公开列表和前台搜索已经支持 `chen`、`陳` 命中“陈”，但详情接口仍只按简体姓氏键名直查，后续小程序可能出现“搜索列表有结果、详情页 404”的割裂体验。
   - 修复：新增 `resolveWorkspaceSurnameKey`，详情接口先按规范姓氏直查，查不到再按拼音和繁体等值解析；无端口逻辑验收覆盖 `name=chen` 与 `name=陳`。

139. 导入框能拒绝拉丁/数字/标点混合姓氏，但服务端快照或浏览器旧缓存恢复时会绕过导入框校验，脏姓氏、脏 Markdown 资料或脏审核项仍可能重新进入前台状态。
   - 修复：`applyWorkspaceSnapshot` 新增统一恢复过滤，服务端快照、本地缓存和导入 JSON 都会过滤无效姓氏、无效 Markdown 条目和无效审核条目；UI 验收覆盖工作区快照恢复过滤。

## 上线建议

1. 生产环境必须设置强 `ADMIN_TOKEN`。
2. `AI_API_KEY` 只放服务端环境变量，不进入前端页面和资料文件。
3. Docker 部署时挂载 `/app/runtime`，保留运行数据和审计日志。
4. 上线前至少执行 `npm run verify:release` 和 `npm run verify:server`。
5. 首批内容建议按 20-50 个高频姓氏扩展 `data/seed-workspace.json`，再接真实数据库。
